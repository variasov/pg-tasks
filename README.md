# pg-tasks

Расширение для PG, добавляющая функционал для управления очередями задач
для внешних процессов.

Вдохновлено [pgmq](https://github.com/pgmq/pgmq).


## Особенности:
- Не имеет внешних зависимостей, только plpgsql
- Не требуются background workers
- Не используется JSON, только нативные типы
- Максимальное переиспользование планов запросов
- Управление задачами из любого клиента БД (создание, перезапуск и т.д.)
- Воркеры (логика) может быть на любом языке


## Клиентские библиотеки:
- [Python](https://github.com/variasov/pg-tasks-python)


## Quickstart

### Установка
Сначала надо установить расширение.

```sql
CREATE SCHEMA tasks;

CREATE EXTENSION pg_tasks SCHEMA tasks;
```

Настоятельно рекомендуется предварительно создать отдельную схему 
для таблиц задач и внутренних объектов расширения. На каждую таблицу-задачу
расширение создает 6 функций и 2 представления. При наличии 100 типов задач
в схеме будет 500 функций и 200 представлений. Поиск объектов вашего приложения
сильно затруднится. С другой стороны, если взять за основу соглашение о том,
что вся схема отдается расширению, и кроме задач там ничего нет,
это не должно быть проблемой.

### Создание очередей (таблиц)

Для каждого типа задач нужно описать свою таблицу. Таблицы создаются по шаблону 
из расширения, и дополняются полями, необходимыми для логики приложения:
```sql
CREATE TABLE tasks.do_something(
    LIKE tasks.template
        INCLUDING DEFAULTS
        INCLUDING IDENTITY
        INCLUDING INDEXES,
    payload text
);
```

Полученная таблица будет выглядеть так:
```sql
CREATE TABLE do_something
(
    id integer GENERATED BY DEFAULT AS IDENTITY 
        (maxvalue 2147483647) PRIMARY KEY,
    
    -- Время создания
    created_at timestamptz DEFAULT now(),  
    
    -- Время, после которого задача должна быть выполнена
    planned_to timestamptz DEFAULT now(),
    
    -- Время, в которое воркер взял задачу на исполнение
    started_at timestamptz,
    
    -- Время, в которое воркер пометил задачу выполненной
    finished_at timestamptz,
    
    -- Количество попыток выполнения
    execution_count integer DEFAULT 0,
    
    -- Комметарий. Поле для свободного использования
    comment text,
    
    -- Пользовательское поле. Их может быть сколько угодно.
    payload text
);
```
Столбцы, нужные для работы расширения нельзя изменять или удалять.

После этого таблицу нужно зарегистрировать в расширении,
указав дополнительные параметры:
```sql
SELECT tasks.register(
    schema_name := 'tasks',
    table_name := 'do_something',
    timeout := '00:01:00',      
    max_retries := 3,           
    storage_time := '24:00:00'
);
```

Расширение создаст в своей схеме функции для работы с задачей:
- `acquire_tasks_do_something()`
- `finish_tasks_do_something()`
- `retry_tasks_do_something()`
- `cancel_tasks_do_something()`
- `reset_tasks_do_something()`
- `clean_tasks_do_something()`

И представление, вычисляющее статистику по задачам:
- `stats_of_tasks_do_something`

Созданные объекты описаны в чистом SQL, потому не требуют особых прав
для учетных записей для воркеров. Также это должно приводить к тому,
что PG соберет обобщенный план запроса для этих функций после нескольких
вызовов, что должно сократить накладные расходы.

### Содание и выполнение задач
Для создания задачи нужно просто вставить новую строку в таблицу для задач:
```sql
INSERT INTO tasks.do_something(payload) VALUES ('hello');
```

После этого, на стороне воркера нужно вызвать функцию acquire_do_something(),
чтобы получить задачи в работу:
```sql
SELECT tasks.acquire_tasks_do_something(1)
```
В ответ вернется одна строка со всеми столбцами из do_something.

После выполнения задачи воркер должен:
- если задача успешно выполнена, то пометить ее завершенной с `finish_<task>()`
- если задачу нельзя выполнить сейчас, но нужно попытаться позднее,
  то запланировать повтор с `retry_<task>()`
- если задачу выполнить невозможно, то отменить задачу с `cancel_<task>()`

Также есть функция `reset_<task>()`, сбрасывающая состояние задачи,
чтобы воркеры могли снова взять ее в работу, и функция `clean_<task>()`,
принимающая дату, и вычищающая таски, старше указанной даты.

## Типы задач
Задачи в воркере можно условно поделить на 2 группы - внутренние и внешние.

Внутренние работают только с данными внутри БД. В таких задачах воркер 
посылает `acquire_<task>()` в режиме автокоммита, затем открывает транзакцию,
делает что-то полезное с данными, затем в **той же транзакции** помечает
задачу выполненной. Так данные точно останутся в констистентном состоянии.

Внешние задачи совершают операции с внешними системами. Запросы во внешние
системы могут занимать продолжительное время, внешние сервисы могут иногда
отвечать с большой задержкой, и во время ожидания ответа воркер не должен
удерживать транзакцию. В таких задачах воркер 
посылает `acquire_<task>()` в режиме автокоммита, затем, получает данные из БД
в режиме автокоммита или транзакцией, **завершает** транзакцию, потом уже
делает запрос во внешнюю систему, затем сохраняет данные и помечает задачу
выполненной в **отдельной** транзакции. 

## Публичные объекты

### tasks.types
Реестр типов задач. Хранит в себе записи о типах задач и их параметрах,
переданных в `tasks.register()` при регистрации.

**!!!WARNING!!!**
**Реестр tasks.types предназначен ТОЛЬКО ДЛЯ ЧТЕНИЯ!**
**!!!WARNING!!!**

### tasks.template
Таблица-шаблон для указания столбцов при создании задач. Не содержит записей,
нужна только ради удобства создания новых таблиц-задач.

### tasks.register
Функция `tasks.register()` нужна для регистрации таблиц-задач в расширении.
Параметры:
- **timeout** - Время, за которое воркер должен успеть выполнить задачу.
  В случае, если воркер, взявший задачу, не успел,
  другой воркер сможет взять задачу на выполнение.
  Если NULL, тогда, в случае, когда воркер взял задачу и повис, сломался,
  либо был перезагружен, другие воркеры не будет пытаться выполнять ее.
  По умолчанию NULL.
- **max_retries** - Максимальное количество попыток выполнения задачи воркером
  в случае ошибок. NULL - повторять, пока задача не будет успешно выполнена.
  По умолчанию NULL.
- **notification_interval** - интервал между NOTIFY при получении новых задач.
  По умолчанию '1s'::interval.
- **storage_time** - Максимальный срок хранения, после которого задача 
  должна быть удалена. NULL - хранить вечно. По умолчанию NULL.

### tasks.unregister
Функция `tasks.unregister()` нужна для удаления таблиц-задач из расширении.
Реального удаления таблиц, столбцов или строк не происходит, только удаление
записей из реестра tasks.types и удаление служебных объектов, связанных с типом
задачи.

Идемпотентна.
